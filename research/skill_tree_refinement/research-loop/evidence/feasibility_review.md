# Feasibility Review

## Current Decision

IDEA-001 minsplit8 has a negative provisional heldout result and minsplit10 is
cancelled. IDEA-002 Phase A has passed implementation review and retrieval
preflight; run its matched dense/routed pair after the active diagnostic exits.
Phase B is feasible but remains contingent on Phase A's frozen gate.

## Native Support

- `gmm_bic.min_split_size` is already accepted by the experiment config/CLI.
- `ProjectedGmmTreeBuilder.cluster_layer` uses it for `too_small` stopping.
- `compute_kmax` also uses it, so selected K must be audited.
- Existing pipeline artifacts already record hierarchy layers, tree state,
  generation token usage, and nodebank outputs.
- Existing rollout/evaluator paths can support future train-side reroll
  validation; reuse must be confirmed before any code change.

## Smallest Credible Path

1. No source edit: generate isolated minsplit8 config from the fresh run.
2. Run build only in a new run directory with the same records and services.
3. Compare the predeclared global L0/upper-layer metrics against minsplit4.
4. Run minsplit10 only under the contract's conditional granularity gate.
5. Select without heldout labels; evaluate at most one setting on full 200, or
   move to the separately contracted VCCS mechanism.

## Main Technical Risks

- Build summaries are stochastic even with low temperature depending on the
  backend; one structural run can be noisy.
- Larger min split can force overly broad communities and generic summaries.
- Reusing an embedding cache is fair only if cache namespace/config fingerprints
  match; otherwise rebuild embeddings.
- Existing dirty source files can alter runtime behavior relative to HEAD; the
  contract therefore requires exact baseline stage-source hashes before launch.
- The current endpoint must be preflighted without proxy leakage and with the
  same model identity used in the run config.

## Phase-B Source-Excluded Admission Preflight

The frozen minsplit4 tree contains 239 L1 cards generated by 101 L0
communities. Source communities contain 1-7 trajectories; the card distribution
is 73 success-only, 72 failure-only, and 94 mixed-outcome cards.

An embedding-only preflight applied the exact retrieval query protocol
(`instruction + Task type`, no answer position) and excluded the union of every
nonzero member task in each card's source community. All 239 cards have both a
source-excluded train-success and train-failure control. The selected controls
cover 52 unique success tasks, 51 unique failure tasks, and 103 unique tasks in
total.

This resolves the validation-unit question without using heldout data. After a
clean isolated train baseline exists, its 200 outputs can be reused as the
without-card side. Phase B then needs at most 478 card-only rerolls, evaluated
with the existing LibreOffice path. Multi-parent support is provenance only;
admission is per unique L1 card and is never double-counted by parent count.

The full deterministic selection ledger is
`evidence/phase_b_source_excluded_preflight.json`. This proves candidate
availability and cost only; it does not establish utility or authorize Phase B
before Phase A's result gate.
